Purpose

The developer layer explains how the static Partner Command Center should evolve into a connected system without confusing demo behavior for production infrastructure.

The current repo is intentionally static: HTML, CSS, JavaScript, mock data, localStorage, iframe widgets, example schemas, and documentation. This makes the partner motion reviewable before authentication, CRM, API, webhooks, and security controls are added.

Implementation principles

  • Keep static demo behavior clearly labeled as demo behavior.
  • Use localStorage only for non-sensitive browser-local state.
  • Do not collect SSNs, bank logins, full tax IDs, account numbers, private documents, or underwriting files in static tools.
  • Do not promise funding, approval, rates, terms, timelines, income, or commissions.
  • Use backend validation, authentication, authorization, audit logging, and secure storage before handling live partner or lead data.
  • Treat schemas and endpoints as contracts, not decorations.

Recommended future build order

  1. Finalize CRM fields and partner status lifecycle.
  2. Create lead submission API with validation and consent capture.
  3. Connect Tally partner signup webhook.
  4. Connect widget inquiry submission to backend API.
  5. Add authentication and role-based access for admin/operator/partner views.
  6. Add webhook signature verification and audit logs.
  7. Add GPT action schema only after backend endpoints exist.
  8. Add production privacy policy and terms matched to real data flows.
